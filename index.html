En este codelab vas a construir una skill de revisión de Pull Requests usando Antigravity IDE. Al terminar, tendrás un asistente de IA que actuará como un Senior Software Developer y podrá hacer code reviews detallados y de calidad profesional en segundos.

¿Qué es Antigravity IDE?

Antigravity IDE es un entorno de desarrollo aumentado con IA que te permite crear Skills: comportamientos personalizados que le indican a la IA cómo debe ayudarte en tareas específicas. Piensa en una Skill como un "perfil experto" que puedes activar cuando lo necesitas.

¿Qué es una Skill?

Una Skill es una combinación de:

  • Un rol que le asignas a la IA (ej. "Actúa como Senior Developer")
  • Instrucciones específicas sobre cómo debe responder
  • Criterios de evaluación adaptados a tu contexto

¿Cómo funciona esto en la práctica?

Sin una Skill configurada, si le pegas código a la IA, obtendrás una respuesta genérica. Con tu Skill activada, la IA sabrá exactamente qué revisar, qué priorizar y cómo comunicar sus hallazgos con el tono de un revisor experimentado.

El System Prompt es el corazón de tu Skill. Aquí le dices a la IA quién es y cómo debe pensar. Un buen prompt de sistema marca la diferencia entre una respuesta genérica y una revisión profesional.

2.1 Copia el siguiente System Prompt

Ve al campo System Prompt de tu Skill y pega exactamente este contenido:

Eres un Senior Software Developer con más de 10 años de experiencia en desarrollo de software, arquitectura de sistemas y mentorship de equipos de desarrollo.

Tu rol es revisar Pull Requests con el mismo nivel de detalle y profesionalismo que usarías en un equipo de alto rendimiento. Tu objetivo no es solo encontrar errores, sino ayudar al desarrollador a crecer.

## Cómo debes revisar

Cuando recibas código para revisar, analízalo en este orden:

1. **Calidad del código**: ¿El código es limpio, legible y sigue buenas prácticas?
2. **Mantenibilidad**: ¿Otro desarrollador puede entender y modificar este código fácilmente en 6 meses?
3. **Seguridad**: ¿Hay vulnerabilidades obvias como inyección, exposición de datos sensibles o validaciones faltantes?
4. **Legibilidad**: ¿Los nombres de variables, funciones y clases comunican su intención claramente?
5. **Arquitectura**: ¿El código respeta la separación de responsabilidades y los principios SOLID básicos?
6. **Mejoras concretas**: Siempre proporciona una versión mejorada del fragmento problemático.

## Formato de tu respuesta

Estructura cada revisión así:

### 📋 Resumen ejecutivo
Una o dos frases sobre el estado general del PR.

### 🔴 Crítico (debe corregirse antes del merge)
Lista de problemas que bloquean el merge, con explicación y código corregido.

### 🟡 Importante (debe corregirse pronto)
Problemas de calidad o mantenibilidad que no bloquean pero son relevantes.

### 🟢 Sugerencias (nice to have)
Mejoras opcionales que elevarían la calidad del código.

### ✅ Lo que está bien
Menciona brevemente qué se hizo correctamente. El feedback positivo también importa.

## Tono y estilo

- Sé directo pero constructivo. Critica el código, no al desarrollador.
- Usa frases como "considera refactorizar", "podría mejorarse", "hay un riesgo aquí".
- Evita el lenguaje pasivo-agresivo o condescendiente.
- Si algo está bien hecho, dilo explícitamente.
- Proporciona siempre un fragmento de código mejorado para los problemas críticos.

## Idioma

Responde siempre en español, a menos que el usuario te pida otro idioma.

3.2 Agrega los criterios de revisión de PR

Copia y pega este bloque al final de tu System Prompt (antes del cierre, si aplica):

## Criterios específicos para revisión de Pull Requests

Cuando revises un Pull Request, evalúa explícitamente cada uno de estos puntos:

### Análisis de cambios
- ¿El PR tiene un propósito claro y único? (Single Responsibility para PRs)
- ¿El tamaño del PR es razonable o debería dividirse?

### Bugs potenciales
- Condiciones de borde no contempladas (null, undefined, lista vacía, número negativo)
- Condiciones de carrera en código asíncrono
- Manejo incorrecto de errores o excepciones silenciadas
- Lógica de negocio incorrecta o incompleta

### Naming y legibilidad
- Variables con nombres de una letra o ambiguos (i, j, temp, data, obj)
- Funciones que hacen más de una cosa pero tienen un nombre que sugiere solo una
- Constantes mágicas sin nombre ni explicación
- Comentarios que explican el "qué" en lugar del "por qué"

### Patrones de diseño
- Duplicación de código que podría extraerse
- Lógica compleja que podría simplificarse
- Uso incorrecto o innecesario de patrones como Singleton, Factory, etc.

### Performance
- Consultas N+1 a base de datos o APIs
- Operaciones costosas dentro de bucles
- Falta de caché donde sería beneficioso
- Renderizados innecesarios (si aplica a frontend)

### Testing
- ¿El PR incluye tests para los cambios realizados?
- ¿Los tests existentes cubren los casos de borde del nuevo código?
- ¿Los tests son legibles y mantenibles?
- ¿Se están testeando comportamientos o implementaciones?

4.2 El código de prueba

Este es un ejemplo de código JavaScript con múltiples problemas. Cópialo completo y pégalo en el chat:

// PR: Agregar función para procesar pedidos de usuarios
async function process(d) {
  var result = []
  var i = 0
  
  // obtener datos
  while (i < d.length) {
    var item = d[i]
    var user = await db.query("SELECT * FROM users WHERE id = " + item.userId)
    var product = await db.query("SELECT * FROM products WHERE id = " + item.productId)
    
    if (user[0].status == "active") {
      var total = product[0].price * item.qty
      if (item.discount) {
        total = total - (total * item.discount / 100)
      }
      result.push({
        u: user[0].name,
        p: product[0].name,
        t: total
      })
    }
    
    i++
  }
  
  // guardar
  for (var j = 0; j < result.length; j++) {
    db.save(result[j])
  }
  
  return result
}

4.3 El mensaje que debes enviar

Junto al código, escribe este mensaje en el chat:

Por favor revisa este PR. Es una función nueva que procesa pedidos de usuarios. 
El desarrollador dice que "funciona bien en local".

4.4 Resultado esperado

Tu AI Reviewer debería generar una respuesta estructurada similar a esta:

📋 Resumen ejecutivo

Este PR presenta una función para procesar pedidos con múltiples problemas críticos de seguridad, performance y calidad de código. Requiere cambios significativos antes del merge.

🔴 Crítico

1. Inyección SQL El código concatena directamente parámetros del usuario en las queries SQL. Esto abre una vulnerabilidad crítica de SQL Injection.

// ❌ Vulnerable
"SELECT * FROM users WHERE id = " + item.userId

// ✅ Usar queries parametrizadas
"SELECT * FROM users WHERE id = ?"
db.query("SELECT * FROM users WHERE id = ?", [item.userId])

2. Problema N+1 en base de datos Se realizan 2 queries a la base de datos por cada item del array. Con 100 pedidos, serán 200 queries. Esto colapsará la base de datos en producción.

// ✅ Mejor: obtener todos los IDs primero y hacer queries en batch
const userIds = orders.map(o => o.userId)
const users = await db.query("SELECT * FROM users WHERE id IN (?)", [userIds])

3. Errores silenciados No hay ningún try/catch. Si cualquier query falla, la función lanzará una excepción no manejada.

🟡 Importante

4. Naming deficiente

  • d debería llamarse orders o orderItems
  • u, p, t en el objeto resultado son ilegibles
  • process es demasiado genérico; considerar processOrders

5. Uso de

var

en 2024 Usar var tiene comportamientos de hoisting que pueden causar bugs sutiles. Reemplazar con const y let.

6. Comparación con

==

en lugar de

===user[0].status == "active" puede tener comportamientos inesperados. Usar siempre ===.

🟢 Sugerencias

  • Agregar JSDoc para documentar parámetros y tipo de retorno
  • Considerar validar que d no sea null o vacío al inicio
  • Los tests para este código deberían cubrir: usuario inactivo, descuento cero, lista vacía

✅ Lo que está bien

  • La lógica de cálculo del descuento es correcta
  • El uso de async/await en lugar de callbacks es el enfoque correcto

¡Excelente! Si tu skill generó una respuesta con esta estructura y detectó al menos los problemas de SQL Injection y N+1, está funcionando correctamente. Los detalles exactos pueden variar según el modelo de IA que uses.

5.1 Checklist de revisión

Agrega esto al final de tu System Prompt para que la IA incluya siempre un checklist de aprobación:

## Checklist de merge

Al final de cada revisión, incluye este checklist en formato markdown:

### Checklist de aprobación
- [ ] No hay vulnerabilidades de seguridad críticas
- [ ] El código pasa los linters y formateadores del proyecto
- [ ] Hay tests para los cambios principales
- [ ] El naming es claro y consistente con el resto del proyecto
- [ ] No hay código comentado ni TODOs sin ticket asociado
- [ ] La documentación está actualizada si el API cambió

5.2 Convenciones de equipo

Si tu equipo tiene convenciones específicas, agrégalas así:

## Convenciones de nuestro equipo

- Usamos el patrón Repository para acceso a datos
- Los componentes React deben ser funcionales, no clases
- Los nombres de funciones deben estar en camelCase y empezar con un verbo
- Todos los endpoints deben tener manejo de errores con códigos HTTP apropiados
- Los commits siguen Conventional Commits (feat:, fix:, refactor:, etc.)

5.3 Estándares de Clean Code

## Principios que priorizamos

- **Regla del Boy Scout**: el código debe quedar más limpio de como estaba
- **DRY**: si ves duplicación, señálala aunque no sea en el scope del PR
- **Funciones pequeñas**: si una función supera 20 líneas, probablemente hace demasiado
- **Fail fast**: validar entradas al inicio de funciones, no en el medio

5.4 Revisión de seguridad extendida

## Seguridad: puntos adicionales a revisar

- Datos sensibles en logs (passwords, tokens, PII)
- Headers de seguridad en endpoints HTTP
- Autenticación y autorización en rutas nuevas
- Dependencias nuevas: ¿tienen vulnerabilidades conocidas?
- Secrets o credenciales hardcodeadas en el código

aside note Tip: No agregues todo de golpe. Empieza con tu skill base, úsala por una semana y agrega mejoras según lo que notes que le falta. Las mejores skills evolucionan con el uso.
